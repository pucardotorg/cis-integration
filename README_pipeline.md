# CIS Pipeline Orchestrator

Runs an ordered list of CIS "use cases" (stages) from a manifest, producing each stage's
output and a final summary. Current V3 order: **1. Filing → 2. Allocation → 3. Court mapping → 4. Objection → 5. Case registration**. Adding a future use case
(notice, proceedings, …) is just a new entry in `pipeline.json`.

```text
uploader/V3/RUN_PIPELINE.sh                             # orchestrator (one command)
uploader/V3/Data/pipeline.json                           # manifest of stages (inputs + config)
uploader/V3/source/cheque_cis_bridge_template.sh         # stage 1: create cases (NACT/138)
uploader/V3/source/allocation_cis_bridge_template.sh     # stage 2: allocate cases to courts
uploader/V3/source/select_court_cis_bridge.sh            # stage 3: court mapping / select active court
uploader/V3/source/case_objection_cis_bridge.sh          # stage 4: objection/scrutiny
uploader/V3/source/registration_cis_bridge.sh            # stage 5: case registration
uploader/V3/output/                                      # all runtime-generated JSON (results, reports, summary)
```

Run scripts stay at the root; worker templates/logic live in `source/`; hand-supplied
JSON (inputs, samples, manifest) lives in `Data/`; everything generated at runtime
(results, intermediate transforms, preflight/postcheck reports, summary) lives in
`output/`. Manifest path fields (`input_file`/`output_file`/`*_script`) are relative
to the uploader root.

## Run

```bash
cd uploader/V3
bash RUN_PIPELINE.sh             # run the pipeline
bash RUN_PIPELINE.sh --validate   # check each stage's input JSON is present (no CIS calls)
```

`--validate` walks every enabled stage and reports whether its input JSON is present
and well-formed (with record count). For stages whose input is generated by a transform
(`transform_from`), it reports PEND and confirms the upstream output exists. Use it before
a real run to confirm you have the inputs for every step. No CIS login, no writes.

## Pipeline flow

```
Drishti filing batch  (carries target_court_no + fmm_case_type per record)
  │  1. STAGE filing    (cheque bridge, file mode)
  ▼
output/DDMMYYYY/<run-id>-filing-cis-results.json
  │
  ├─ TRANSFORM transform_filing_to_allocation.py
  ▼
Data/allocation-input.json
  │  2. STAGE allocation
  │     preflight  ->  submit (formaction=8)  ->  postcheck
  ▼
output/DDMMYYYY/<run-id>-allocation-cis-allocation-results.json
  │
  ├─ TRANSFORM transform_allocation_to_select_court.py
  ▼
Data/select-court-input.json
  │  3. STAGE court mapping / select_court
  ▼
output/DDMMYYYY/<run-id>-select_court-cis-select-court-results.json
  │
  ├─ TRANSFORM transform_filing_to_case_objection.py  (uses filing output)
  ▼
Data/case-objection-input.json
  │  4. STAGE objection / scrutiny
  ▼
output/DDMMYYYY/<run-id>-case_objection-cis-case-objection-results.json
  │
  ├─ TRANSFORM transform_case_objection_to_registration.py
  ▼
Data/registration-input.json
  │  5. STAGE case registration
  ▼
output/DDMMYYYY/<run-id>-registration-cis-registration-results.json
  ▼
output/DDMMYYYY/<run-id>-pipeline-summary.json  +  printed summary table
```

Each stage: optional **transform** (from a prior stage output) → optional **preflight**
(read-only gate; errors abort the stage by default) → **bridge script** → optional
**postcheck** → per-stage summary. Each stage writes its own output file so you can
inspect/retry independently.

### Adding a future use case

Append a stage object. Fields:

| field | meaning |
|---|---|
| `name` | unique stage id; referenced by `transform_from` |
| `run` | `true`/`false` to skip |
| `input_file` / `output_file` | relative to the uploader dir |
| `bridge_script` | worker `.sh` (file mode: reads `INPUT_JSON`/`OUTPUT_JSON` env) |
| `preflight_script` | optional; called as `<script> --preflight <input> <report>` |
| `postcheck_script` | optional; called as `<script> --postcheck <output> <report>` |
| `transform_from` | optional; name of upstream stage whose output feeds this stage |
| `transform_script` | optional; `python3 <script> <src> <dst> [summary]` |

## Where `target_court_no` and `allocation_dt` come from (Option A)

The filing input JSON carries `target_court_no`, `fmm_case_type`, and `allocation_dt` per
record. The cheque bridge echoes all three through in `cis-results.json`.
`transform_filing_to_allocation.py` copies them into the allocation input. One source of
truth — no separate court-assignment map.

`allocation_dt` is per-record and respected as-is, so a case can be allocated on a
different date than it was created. It defaults to today only if absent.

## Self-healing login

Both bridges share the same hardened `login_cis()`: if CIS returns `output=='UserLogged'`,
they call `index_unlock.php` → `loginajax1.php` `x=checkunlock_fromindex` to unlock, then
retry the login. So a crashed prior run (or a human) leaving the user logged in won't block
the pipeline.

## Summary

`pipeline-summary.json` (and stdout table) shows per stage: input count, output
ok/total/fail, preflight ok/error, postcheck ok/mismatch.

## Files written by a run (under `output/DDMMYYYY/`)

Each run writes to the current day folder and prefixes related files with a shared run id
(`HHMMSS-rand`) so runs never overwrite each other and remain auditable.

```text
output/DDMMYYYY/<run-id>-filing-cis-results.json
output/DDMMYYYY/<run-id>-allocation-cis-allocation-results.json
output/DDMMYYYY/<run-id>-allocation-preflight-report.json
output/DDMMYYYY/<run-id>-allocation-postcheck-report.json
output/DDMMYYYY/<run-id>-select_court-cis-select-court-results.json
output/DDMMYYYY/<run-id>-case_objection-cis-case-objection-results.json
output/DDMMYYYY/<run-id>-registration-cis-registration-results.json
output/DDMMYYYY/<run-id>-pipeline-summary.json
output/DDMMYYYY/<run-id>-run-manifest.json
```

Individual stages are run with `RUN_STAGE.sh`; their input/output/bridge paths are read
from `Data/pipeline.json` and outputs use the same `output/DDMMYYYY/<run-id>-...` pattern.
